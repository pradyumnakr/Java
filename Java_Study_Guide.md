# Java Study Guide

## JVM (Java Virtual Machine)
The JVM is the core engine responsible for running Java bytecode. It performs the following essential functions:
- **Loads** `.class` files (compiled bytecode)
- **Verifies** the bytecode for security and structural correctness
- **Converts** bytecode into underlying machine code
- **Executes** the resulting machine code
- **Manages memory**, organizing the Heap and Stack
- **Handles Garbage Collection** to automatically free up unused memory

> **🔥 Key Point:** The JVM is what makes Java **platform-independent**. You compile the code once, and the JVM runs it anywhere regardless of the underlying operating system.

## JRE (Java Runtime Environment)
The JRE is the complete software environment required to run Java applications. It consists of:
- The **JVM** (to execute the code)
- **Core libraries** (e.g., `java.lang`, `java.util`)
- Additional **runtime support files** required for execution

> **🔥 Key Point:** The JRE is used strictly to **run** Java programs. You do **NOT** need the JDK (Java Development Kit) just to execute a program; the JRE alone is completely sufficient.

## JDK (Java Development Kit)
The JDK is the complete software development environment required for developing Java applications. It combines the **JRE** with essential **development tools**, including:
- The **JRE** (Java Runtime Environment)
- The **Compiler** (`javac`) to turn source code into bytecode
- A **Debugger** to test and fix code
- The **Javadoc** tool to auto-generate documentation
- The **Jar** tool to package applications

> **🔥 Key Point:** The JDK is strictly used to **develop** Java programs. If you are writing and compiling Java code, you must have the JDK installed.

## The Big Picture (JDK vs. JRE vs. JVM)
You can think of their relationship as: **JDK ⊃ JRE ⊃ JVM**.

**A helpful analogy:**
- 🧠 **JVM** = The Engine
- 🚗 **JRE** = The Car (engine + fuel + parts to run)
- 🏭 **JDK** = The Car Factory (tools to build the car)

## Bytecode
**Bytecode** is the intermediate, platform-independent code generated by the Java compiler (`javac`) from `.java` source files.
- It is executed by the **JVM**, which converts it into machine code for the underlying operating system.

## The Java Compilation and Execution Lifecycle

### 1. Write Source Code
First, you write your Java code in a simple text file and save it with a `.java` extension (*e.g., `Main.java`*).
```java
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}
```

### 2. Compile Code (`javac Main.java`)
What happens during compilation:
- The `javac` compiler (which is part of the **JDK**) takes the source code.
- It converts it into **bytecode**.
- It generates a new file called `Main.class`.

> 👉 **Remember:** This `.class` bytecode file is platform-independent.

### 3. Execute Code (`java Main`)
What happens during execution:
- The **JVM** kicks in.
- The **Class Loader** loads the `Main.class` file into memory.
- The **Bytecode Verifier** checks the code for security issues and structural correctness.
- The **Execution Engine** processes the bytecode.

Inside the Execution Engine:
- The **Interpreter** reads and runs the bytecode line by line.
- The **JIT (Just-In-Time) Compiler** identifies frequently used code segments (hotspots) and compiles them directly into faster machine code.
- The **Garbage Collector** continuously runs in the background to manage memory (cleaning up unused objects).

> 👉 Finally, the resulting raw **machine code** is executed directly on your operating system's hardware.

### Overarching Execution Flow
```text
Main.java                  [Source Code]
   ↓ (javac)
Main.class                 [Bytecode]
   ↓ (JVM loads)
Interpreter / JIT Compiler [Execution Engine]
   ↓
Machine Code               [Hardware-Level Code]
   ↓
Output on Screen           [Execution Result]
```

## Data Types in Java
In Java, data types define the type of data a variable can store and the operations allowed on it.

Java has two main categories of data types:

### 1️⃣ Primitive Data Types (8 types)
These are basic built-in types. They store actual values:

1. **`byte`**
   - **Size**: 1 byte (8 bits)
   - **Range**: -128 to 127
   - **Use Case**: Used to save memory in large arrays
   - **Example**: `byte age = 25;`
2. **`short`**
   - **Size**: 2 bytes (16 bits)
   - **Range**: -32,768 to 32,767
   - **Example**: `short temperature = 30000;`
3. **`int`**
   - **Size**: 4 bytes (32 bits)
   - **Range**: -2^31 to 2^31 - 1
   - **Use Case**: Default integer type in Java
   - **Example**: `int salary = 50000;`
4. **`long`**
   - **Size**: 8 bytes (64 bits)
   - **Use Case**: Used for very large numbers
   - **Details**: Must end with `L`
   - **Example**: `long population = 7800000000L;`
5. **`float`**
   - **Size**: 4 bytes (32 bits)
   - **Details**: Single precision decimal; must end with `f`
   - **Example**: `float price = 99.99f;`
6. **`double`**
   - **Size**: 8 bytes (64 bits)
   - **Details**: Higher precision than float; default decimal type in Java
   - **Example**: `double pi = 3.1415926535;`
7. **`char`**
   - **Size**: 2 bytes (16 bits)
   - **Details**: Stores a single character; uses Unicode
   - **Example**: `char grade = 'A';`
8. **`boolean`**
   - **Size**: JVM dependent
   - **Details**: `true` or `false`
   - **Example**: `boolean isJavaFun = true;`

### 2️⃣ Non-Primitive Data Types (Reference Types)
Unlike primitive types which store the actual value, **non-primitive (reference) types** store the memory *address* (or reference) where the data is located. These include:

1. **`String`** 
   - **Details**: Not primitive — it's actually a built-in class in Java.
   - **Example**: `String name = "Pradyumna";`
2. **Arrays** 
   - **Details**: A collection of similar data types.
   - **Example**: `int[] numbers = {1, 2, 3, 4};`
3. **Class / Object**
   ```java
   class Student {
       String name;
       int age;
   }
   ```
4. **Interfaces**

## Type Casting in Java
**Type casting** means converting one data type into another. 

There are two main types:

### 🔹 A) Widening (Implicit Casting)
- **What it is:** Smaller type → Larger type
- **How it works:** Done automatically by Java (because there is no risk of data loss).
- **Order:** `byte` → `short` → `int` → `long` → `float` → `double`

**Example:**
```java
int a = 10;
double b = a;  // int → double (automatic)
```
- ✅ No data loss
- ✅ Safe conversion

### 🔹 B) Narrowing (Explicit Casting)
- **What it is:** Larger type → Smaller type
- **How it works:** Must be done manually because compiling it implicitly would pose a risk of data loss.

**Example:**
```java
double x = 10.75;
int y = (int) x;  // explicit cast
System.out.println(y);  // 10
```
- ⚠ Decimal part is lost
- ⚠ Possible data loss

Java provides object-oriented "wrapper" class versions of all primitive types.

| Primitive | Wrapper Class |
| --- | --- |
| `byte` | `Byte` |

### Why use Wrapper Classes?
- **Collections:** Data structures like `ArrayList` can only store objects, not primitive types.
- **Generics:** Necessary when working with Java generics (`List<Integer>`, not `List<int>`).
- **Utility Methods:** They provide built-in methods for data conversion and manipulation.

**Example:**
```java
int a = 10;
Integer obj = Integer.valueOf(a);
```

**Useful Methods:**
```java
Integer.parseInt("123");   // Converts String to int
Integer.toString(100);     // Converts int to String
Integer.MAX_VALUE;         // Returns the maximum possible int value
```

| Primitive | Wrapper |
| --- | --- |
| Faster execution | Slightly slower |
| Cannot be `null` | Can be `null` |
| No methods | Has methods |
| Stored in **Stack memory** | Stored in **Heap memory** |

## Autoboxing & Unboxing
*Introduced in Java 5.*

### 🔹 Autoboxing
Autoboxing is the automatic conversion that the Java compiler makes between the primitive types and their corresponding object wrapper classes.
- **What it is:** Primitive → Wrapper automatically.

**Example:**
```java
int a = 10;
Integer obj = a;  // Autoboxing (Done automatically by the compiler)
```

**Internally, Java executes this as:**
```java
Integer obj = Integer.valueOf(a);
```

### 🔹 Unboxing
Unboxing is the reverse process — converting from the wrapper class object back to the corresponding primitive type.
- **What it is:** Wrapper → Primitive automatically.

**Example:**
```java
Integer obj = 20;
int a = obj;  // Unboxing (Done automatically by the compiler)
```

**Internally, Java executes this as:**
```java
int a = obj.intValue();
```

## What Happens on Integer Overflow?
Java does **NOT** throw an error on overflow. Instead, it silently wraps around to the lowest possible value.

**🔹 Example:**
```java
int x = Integer.MAX_VALUE;
System.out.println(x);        // 2147483647

x = x + 1;
System.out.println(x);        // -2147483648
```

## Memory Management (Stack vs. Heap Memory)
Java memory is mainly divided into two areas:
1. **Stack Memory:** Stores method calls and local variables.
2. **Heap Memory:** Stores all Objects and instance variables.

### 🔹 The Stack
The Stack is responsible for storing:
- Primitive variables
- Reference variables (the memory address)
- Method calls and Stack frames

**Characteristics of the Stack:**
- Faster access
- Thread-specific (each thread has its own stack)
- Automatically cleaned and destroyed after the method ends

### 🔹 The Heap
The Heap is responsible for storing:
- Objects
- Instance variables
- Arrays

**Characteristics of the Heap:**
- Shared across all threads globally
- Managed by Garbage Collector
- Much larger than the stack

**Example:**
```java
int a = 10;
Student s = new Student();
```
- `a` → The primitive variable is stored directly in the Stack.
- `s` → The reference variable (memory address) is stored in the Stack.
- `new Student()` → The actual Object, including its instance variables, is stored in the Heap.

### 🆚 Stack vs. Heap Memory

| Feature | Stack Memory | Heap Memory |
| --- | --- | --- |
| **Stores** | Local variables & method frames | Objects & Instance variables |
| **Speed** | Faster | Slower |
| **Scope** | Thread-specific | Shared globally across threads |
| **Cleanup** | Auto cleanup (when method ends) | Managed by Garbage Collector (GC) |

## Pass by Value in Java
👉 **Java is strictly pass-by-value. Even for objects.**

### 🔹 Primitive Example
When passing a primitive type, Java passes a **copy** of the value. Modifying the copy inside the method has no effect on the original variable.

```java
public void change(int x) {
    x = 100;
}

int a = 10;
change(a);
System.out.println(a); // Output is 10
```
**Why?** Because only a copy of the actual value `10` was passed into the method `change()`.

## String vs. StringBuilder

### 🔹 String
- **Immutability:** Strings are **immutable** (cannot be changed once created).
- **Storage:** Stored in the **String Pool** (a special memory region inside the Heap).
- **Modification behavior:** Every time you modify a String, it actually creates an entirely new object in memory.

**Example:**
```java
String s = "Hello";
s = s + " World";  // This operation actually creates a new object in the String Pool
```

### 🔹 StringBuilder
- **Mutability:** `StringBuilder` is **mutable** (can be changed).
- **Performance:** Much faster for operations requiring a lot of string modifications.
- **Thread Safety:** Not thread-safe (unlike `StringBuffer`).
- **Modification behavior:** Modifies the existing object instead of creating a new one.

**Example:**
```java
StringBuilder sb = new StringBuilder("Hello");
sb.append(" World");  // Modifies the existing object; no new object is created
```

### 🆚 Interview Table: String vs. StringBuilder
| Feature | `String` | `StringBuilder` |
| --- | --- | --- |
| **Mutability** | Immutable | Mutable |
| **Modification Speed** | Slower | Faster |
| **Thread Safety** | Thread-safe | Not thread-safe |

### 🤔 Why is String Immutable?
- **Security:** Strings are frequently used for sensitive data like credentials or database URLs. If mutable, these could be tampered with.
- **Thread Safety:** Immutable objects can be safely shared among multiple threads without needing complex synchronization.
- **String Pool Optimization:** Because Strings cannot be changed, Java can safely use the String Pool to have multiple references point to the exact same memory location, saving massive amounts of memory.
- **Hashcode Caching:** Since the value can never change, the hashcode of a String is calculated once and cached forever. This makes them perfectly optimized for use as keys in a `HashMap`.

## Java Memory Model (JMM)
The Java Memory Model defines **how threads interact through memory**.

### 🔹 Key Concepts
- **Stack:** Each thread has its own isolated stack memory.
- **Heap:** The heap is shared across all threads.
- **Visibility Issues:** Because threads can cache variables in their own local memory (registers/L1 cache), changes made by one thread might not be immediately visible to another.

### ⚠️ Visibility Problem Example
```java
boolean flag = false;

// Thread 1 updates the shared variable:
flag = true;

// Thread 2 is waiting for the flag to change:
while(!flag) { 
    // This loop might run forever! 
}
```
**What happens here?** 
Thread 2 might have cached the `flag` as `false` in its local memory. Even though Thread 1 changed it to `true` in the main heap memory, Thread 2 **may never see the update**, leading to an infinite loop.

**✅ The Fixes:**
There are several ways to ensure visibility across threads:
1. **Using `volatile`**: Forces the JVM to read/write the variable directly from main memory (Heap), bypassing local thread caches. Example: `volatile boolean flag;`
2. **Using `synchronized` blocks**: Ensures that changes made by one thread are visible to others when they enter the synchronized block.
3. **Using Atomic variables**: Classes like `AtomicBoolean` or `AtomicInteger` manage synchronization internally.

### 🛡️ JMM Guarantees
The Java Memory Model makes three primary guarantees regarding concurrent operations:
1. **Atomicity:** Operations that execute entirely as a single unit or not at all (e.g., reading/writing reference variables and most primitive variables).
2. **Visibility:** Changes made by one thread to shared data are visible to other threads (achieved via `volatile`, `synchronized`, etc.).
3. **Ordering:** The compiler/JVM may reorder instructions for optimization, but JMM guarantees that within a thread, the execution happens as-if sequentially (Happens-Before relationship).

## Java Generics
*Introduced in Java 5.*
Generics allow you to add **compile-time type safety** to a system. It ensures that collections and classes operate on specific object types, preventing bugs during runtime.

### 🔹 Without Generics
When not using generics, you have to cast objects when retrieving them from a collection:
```java
ArrayList list = new ArrayList();
list.add("Hello");

// Needs explicit casting to (String)
String s = (String) list.get(0);
```

### 🔹 With Generics
With generics, you explicitly declare the type. The compiler enforces checking, and no manual casting is required:
```java
ArrayList<String> list = new ArrayList<>();
list.add("Hello");

// No casting needed
String s = list.get(0);
```

## Class in Java
A **class** is a blueprint or template used to create objects.
> 👉 *A class does **NOT** occupy memory until an object is created.*

It can contain:
- **Fields** (variables)
- **Methods** (functions / behavior)
- **Constructors**
- **Static members**
- **Nested classes**

### 🏗 Real-Life Analogy
- **Class** = Blueprint of a House
- **Object** = Actual House built from that blueprint

### 🔹 Example
```java
class Student {
    String name;   // field (instance variable)
    int age;

    void study() { // method (behavior)
        System.out.println(name + " is studying");
    }
}
```
**Here:**
- `Student` → The Class
- `name`, `age` → Instance variables
- `study()` → Behavior

## Object in Java
An **object** is an instance of a class. It represents a real-world entity that is built using the class template.

### 🧩 What makes up an Object?
1. **State:** Represented by data/attributes (instance variables).
2. **Behavior:** Represented by methods (functions).
3. **Identity:** Represented by a unique memory address assigned by the JVM.

### 📌 Creating an Object
```java
Student s1 = new Student();
```

**What happens internally?**
- `new Student()` → The actual Object data (variables and methods) is created and stored in the **Heap** memory.
- `s1` → The resulting reference variable (holding the memory address) is stored in the local **Stack** memory.

## Interface in Java
An **interface** is a contract. It defines *what* a class using it must do, but not *how* it does it.

### 📌 Basic Interface Example
First, define the interface:
```java
interface Animal {
    void makeSound();
}
```

Now, a class can implement it. The class is forced to define *how* `makeSound()` works:
```java
class Dog implements Animal {
    public void makeSound() {
        System.out.println("Bark");
    }
}
```

### 🤔 Why Use Interfaces?
- Achieve **abstraction** (hiding implementation details)
- Achieve **multiple inheritance** (since Java doesn't allow extending multiple classes)
- Maintain **loose coupling** between components
- Encourage **better design** and clean architecture

### 🔥 Multiple Inheritance Example
Java allows a class to implement **multiple interfaces**, but it can only extend *one* class.

```java
interface Flyable {
    void fly();
}

interface Swimmable {
    void swim();
}

// A Duck can be both Flyable and Swimmable!
class Duck implements Flyable, Swimmable {
    public void fly() {
        // Flying logic
    }
    public void swim() {
        // Swimming logic
    }
}
```

### 🆚 Class vs. Interface

| Feature | Class | Interface |
| --- | --- | --- |
| **Implementation** | ✅ Yes, can have logic | ❌ No (before Java 8) |
| **Multiple Inheritance** | ❌ No | ✅ Yes |
| **Constructors** | ✅ Yes | ❌ No |
| **Fields** | Typical instance variables | Only `public static final` fields |
| **Object Creation** | ✅ Yes | ❌ No |

### 🚀 Interfaces After Java 8
Starting with Java 8, interfaces were greatly enhanced to allow certain types of concrete implementations:
- **Default Methods:** Methods with a body, using the `default` keyword.
- **Static Methods:** Class-level methods that belong strictly to the interface.

**Example:**
```java
interface Vehicle {
    default void start() {
        System.out.println("Starting...");
    }
}
```

### 🦸‍♂️ Why an Interface is Powerful
One of the core reasons modern Java developers use interfaces is because it enables **Polymorphism**.

```java
// Our interface acting as a contract
interface Animal {
    void makeSound();
}

// Our concrete implementations
class Dog implements Animal { ... }
class Cat implements Animal { ... }
```

**Because of Polymorphism, we can write:**
```java
Animal a = new Dog();
a.makeSound();

Animal b = new Cat();
b.makeSound();
```
Now, our code depends directly on `Animal`, and not explicitly on `Dog` or `Cat`! 

> 💡 This is a powerful, standard core principle known as: **"Programming to an interface, not an implementation."**

### 🔗 It Reduces Coupling (Very Important in Interviews)
Interfaces are critical for reducing coupling between classes.

**Without Interface (Tightly Coupled):**
```java
Dog dog = new Dog(); // Code is tightly coupled to the Dog class
```

**With Interface (Loosely Coupled):**
```java
Animal animal = new Dog(); // Depends only on the abstract Animal interface
```
With the loosely coupled approach, you can later replace `Dog` with `Cat` seamlessly without having to change any other code!

**Method Parameter Example:**
```java
void playSound(Animal animal) {
    animal.makeSound();
}
```
Now this single method works for `Dog`, `Cat`, `Cow`, `Lion`, and any future `Animal` you create! Without the interface, you’d be forced to write hardcoded, duplicate methods for every single animal (e.g., `void playSound(Dog dog)`, `void playSound(Cat cat)`, etc.).

> 🚨 **To summarize:** An Interface enforces a strict rule. It acts as a **contract** that states: *"Anything that is an Animal MUST be able to `makeSound()`"*.

### 🌟 Interfaces Help With:
- **Polymorphism**
- **Abstraction**
- **Loose Coupling**
- **Testability** (ease of mocking)
- **Dependency Injection**
- **Clean Architecture**

### ❌ What Happens If You Don’t Use an Interface?
Without an interface, your code becomes:
- **Tightly coupled** (Hardcoded dependencies)
- **Hard to scale** (Adding new functionality leads to massive code rewrites)
- **Hard to test** (Cannot easily mock out dependencies for unit tests)
- **Hard to replace implementations** (Requires altering existing concrete code rather than just adding a new implementing class)

### 📌 Interface Variables (`public static final`)

If you declare a variable inside an interface:
```java
interface Animal {
    int AGE = 10;
}
```
Java **automatically** treats it as this, even if you don't write it:
```java
public static final int AGE = 10;
```

**What Does That Mean?**
- `public` → Accessible from everywhere.
- `static` → Belongs to the interface itself, not any object instance.
- `final` → It is a constant and cannot be changed.

**❓ Why is it strictly this way?**
Because an interface simply represents a **contract**.
- It **cannot** hold object state.
- It **cannot** have instance variables.
- It is strictly meant to define behavior, not state.

**Example Usage:**
```java
interface Constants {
    int MAX_VALUE = 100;
}

// Usage elsewhere in the code
System.out.println(Constants.MAX_VALUE);
```

## Default Methods (Java 8 Feature)
### ❓ What is `default` in an Interface?
The `default` keyword allows an interface to provide a method implementation.

- **Before Java 8:** Interfaces could only have method declarations. No method body was allowed.
- **After Java 8:** You can write a method with a body directly inside the interface using the `default` keyword.

### 🔹 Example
```java
interface Animal {
    void makeSound(); // Abstract method

    // Default method (has an implementation body)
    default void sleep() {
        System.out.println("Sleeping...");
    }
}
```

**Here:**
- `makeSound()` → **Must** be implemented by the class.
- `sleep()` → Is **already implemented** and inherited for free!

### 🤔 What Does `default` Actually Do?
It provides a default behavior to all classes that implement the interface! Any class implementing `Animal` automatically gets the `sleep()` method.

**Now, the implementing class only needs to define the abstract methods:**
```java
class Dog implements Animal {
    public void makeSound() {
        System.out.println("Bark");
    }
}
```

Because `Dog` automatically inherited `sleep()`, you can call it freely:
```java
Dog d = new Dog();
d.makeSound(); // Prints: Bark
d.sleep();     // Prints: Sleeping...
```

### 🕰️ Why Were Default Methods Introduced?
The primary reason default methods were introduced was to support **Backward Compatibility**.

**Imagine this situation:**
Java has an existing core interface:
```java
interface List {
    void add(E e);
}
```

**Now, Java wants to add a new `sort()` method:**
- **If they added a normal abstract method:** Every single class implementing `List` worldwide would instantly break because they don't have an implementation for `sort()`.

**Instead, they do this:**
```java
default void sort() {
    // implementation
}
```
**Now:**
- Existing classes **still work**.
- There are **no compilation errors**.
> 🌟 This is the main reason `default` exists!

### 💡 What Happens If Class Overrides It?
The class can override the default method.

```java
class Dog implements Animal {
    public void makeSound() {
        System.out.println("Bark");
    }

    // Overriding the default method
    public void sleep() {
        System.out.println("Dog sleeping");
    }
}
```
> 👉 **Rule:** The class method *always* wins over the default method!

### 🚨 Important Rules for Default Methods
1. **Inherited:** Default methods are inherited by the implementing class.
2. **Can be overridden:** The implementing class can choose to provide its own logic.
3. **Multiple inheritance conflicts:** If two interfaces provide the exact same default method and a class implements both, the class **must** explicitly override the method to resolve the conflict (The Diamond Problem).
4. **No instance variables:** They cannot access instance variables, because interfaces themselves do not hold state.

### ⚙️ Static Methods in Interface
Java 8 also allows `static` methods inside of an interface. Just like standard static methods, these belong to the interface itself, not an implementing object.
**🔹 Example:**
```java
interface MathUtils {
    static int add(int a, int b) {
        return a + b;
    }
}
```

**How to call it:**
```java
MathUtils.add(10, 20); // Correct: Called directly on the interface name
```

### 🚨 Important Rules for Static Interface Methods
1. **Cannot be overridden:** Implementing classes cannot override static methods.
2. **Not inherited:** They are not inherited by the implementing class.
3. **Must use interface name:** You *must* call them using the interface name itself.

**Example:**
```java
Dog.add();         // ❌ Wrong (Not inherited by implementing class)
MathUtils.add();   // ✅ Correct (Called directly on the interface name)
```

### 🤔 Why Static Methods in Interfaces?
Static methods were added to interfaces primarily to provide:
1. **Utility/Helper Methods:** Methods that can help interface classes without requiring a separate utility class.
2. **Method Grouping:** The ability to group related helper methods directly inside the interfaces they belong to.
3. **Cleaner API Design:** Leads to improved cohesion and encapsulation.

## The Diamond Problem with Default Methods
*⭐ Interview Favorite*

If an implementing class inherits from multiple interfaces that contain `default` methods with the exact same signature, Java won't know which default implementation to choose!

### 🔹 The Problem Setup
```java
interface A {
    default void show() { System.out.println("A"); }
}

interface B {
    default void show() { System.out.println("B"); }
}

class C implements A, B {
    // ❌ ERROR: Java doesn't know whether to use A's show() or B's show()
}
```

### ✅ The Solution
To fix this, the implementing class **must** explicitly override the conflicting method and resolve the ambiguity. You can call a specific interface's default method using the `InterfaceName.super.methodName()` syntax:
